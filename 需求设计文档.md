## 1. 设计目标与原则

### 1.1 设计目标
- 将《需求文档.md》中定义的 9 项功能需求转化为可落地的技术方案。
- 明确模块边界、数据结构、关键算法、状态流转与接口契约。
- 为后续编码实现、测试与部署提供统一依据。

### 1.2 设计原则
1. **关注点分离**：棋盘逻辑、胜负判定、AI 算法、UI 渲染、状态管理各自独立模块，互不耦合。
2. **纯函数优先**：核心逻辑（落子校验、胜负判定、AI 评分）设计为纯函数，无副作用，便于测试。
3. **单文件可测**：关键算法不依赖 DOM 与框架，可在 Node 环境直接单测。
4. **零后端依赖**：所有状态驻留内存，偏好类信息持久化至 localStorage。
5. **类型安全**：全量 TypeScript，核心数据结构以类型别名 / 接口显式约束。

## 2. 总体架构

### 2.1 架构分层
系统采用四层结构，自下而上依次为：

```
┌─────────────────────────────────────────────┐
│           表现层 (UI Layer)                 │  React 组件 / 样式
├─────────────────────────────────────────────┤
│         状态管理层 (State Layer)            │  Zustand Store
├─────────────────────────────────────────────┤
│         业务逻辑层 (Logic Layer)            │  落子 / 判定 / 回合
├─────────────────────────────────────────────┤
│         算法层 (Algorithm Layer)            │  AI 评分 / 搜索
└─────────────────────────────────────────────┘
```

- **算法层**：无状态纯函数，不引用任何框架 API。
- **业务逻辑层**：组合算法层能力，封装对局规则，产出新棋盘状态。
- **状态管理层**：集中托管对局状态，响应 UI 事件并调用逻辑层。
- **表现层**：订阅状态、渲染界面、捕获用户事件并派发动作。

## 3. 核心数据结构

### 3.1 全局类型定义（`types/index.ts`）

```typescript
// 棋子类型：0=空 1=黑 2=白
export type Stone = 0 | 1 | 2;

// 坐标（行、列，0~14）
export interface Coord {
  row: number;
  col: number;
}

// 棋盘：15×15 二维数组
export type Board = Stone[][];

// 玩家方
export type Player = 1 | 2;          // 1=黑 2=白
export type Side = 'black' | 'white';

// 游戏模式
export type GameMode = 'pvp' | 'pve'; // 人人对战 / 人机对战

// AI 难度
export type Difficulty = 'easy' | 'normal' | 'hard';

// 对局状态
export type GameStatus =
  | 'idle'        // 未开始
  | 'playing'     // 进行中
  | 'won'         // 已分胜负
  | 'draw'        // 和棋
  | 'resigned';   // 认输结束

// 单步历史记录
export interface MoveRecord {
  player: Player;
  coord: Coord;
  stepNo: number;     // 第几手，从 1 开始
  timestamp: number;  // 落子时间戳
}

// 胜负判定结果
export interface WinResult {
  winner: Player | null;     // null 表示未分胜负
  line: Coord[];             // 连珠的坐标序列
  direction: Direction | null;
}

export type Direction = 'horizontal' | 'vertical' | 'diagonalDown' | 'diagonalUp';

// 对局配置
export interface GameConfig {
  mode: GameMode;
  humanSide: Side;           // 人机模式下人类执子方
  difficulty: Difficulty;
}

// 对局状态聚合（Store 主态）
export interface GameState {
  config: GameConfig;
  board: Board;
  history: MoveRecord[];
  currentTurn: Player;       // 当前回合方
  status: GameStatus;
  winResult: WinResult | null;
  startTime: number | null;
  endTime: number | null;
}
```

### 3.2 棋盘常量（`constants/board.ts`）

```typescript
export const BOARD_SIZE = 15;          // 15×15
export const WIN_COUNT = 5;            // 五子连珠
export const DIRECTIONS: ReadonlyArray<[number, number, Direction]> = [
  [0, 1, 'horizontal'],    // 横向
  [1, 0, 'vertical'],      // 纵向
  [1, 1, 'diagonalDown'],  // 正斜 ↘
  [1, -1, 'diagonalUp'],   // 反斜 ↙
];
```

## 4. 核心算法设计

### 4.1 棋盘初始化与落子校验（`logic/board.ts`）

```typescript
// 创建空棋盘
function createEmptyBoard(): Board;

// 判断坐标是否在棋盘范围内
function isInRange(coord: Coord): boolean;

// 判断指定位置是否可落子（在范围内且为空）
function canPlace(board: Board, coord: Coord): boolean;

// 落子（不可变更新，返回新棋盘）
function placeStone(board: Board, coord: Coord, player: Player): Board;
```

**设计要点**：
- `placeStone` 采用不可变更新（拷贝新行数组），保证 React 状态可比对、便于悔棋回放。
- 所有函数为纯函数，无副作用。

### 4.2 胜负判定（`logic/referee.ts`）

```typescript
// 判定最后一次落子是否形成五连珠
function checkWin(board: Board, lastMove: Coord): WinResult;

// 判定是否和棋（棋盘已满且无人获胜）
function checkDraw(board: Board): boolean;
```

**算法说明**：
- 仅以最新落子点为中心，沿 4 个方向（横、竖、正斜、反斜）双向延伸统计同色连子数。
- 时间复杂度 O(1)（与棋盘大小无关，仅与 WIN_COUNT 相关），远优于全盘扫描。
- 返回的 `line` 为连珠坐标序列，用于 UI 高亮。

**伪代码**：
```
checkWin(board, last):
  player = board[last.row][last.col]
  if player == 0: return { winner: null, line: [], direction: null }
  for each (dr, dc, dir) in DIRECTIONS:
    count = 1
    line = [last]
    // 正向延伸
    (r, c) = (last.row + dr, last.col + dc)
    while inRange(r,c) and board[r][c] == player:
      count++; line.push((r,c)); r += dr; c += dc
    // 反向延伸
    (r, c) = (last.row - dr, last.col - dc)
    while inRange(r,c) and board[r][c] == player:
      count++; line.unshift((r,c)); r -= dr; c -= dc
    if count >= WIN_COUNT:
      return { winner: player, line, direction: dir }
  return { winner: null, line: [], direction: null }
```

### 4.3 棋谱历史管理（`logic/history.ts`）

```typescript
// 追加一步
function appendMove(history: MoveRecord[], move: MoveRecord): MoveRecord[];

// 悔棋 N 步（N 由调用方决定：PVP=1, PVE=2）
function undoMoves(history: MoveRecord[], n: number): MoveRecord[];

// 根据历史重建棋盘（用于悔棋后还原）
function rebuildBoard(history: MoveRecord[]): Board;
```

**设计要点**：
- 悔棋后通过 `rebuildBoard` 从剩余历史重建棋盘，避免维护多份棋盘快照。
- PVE 模式悔棋两步保证回合仍归玩家。

### 4.4 回合切换（`logic/turn.ts`）

```typescript
function nextTurn(current: Player): Player;   // 1 -> 2, 2 -> 1
function isFirstMove(history: MoveRecord[]): boolean;
```

## 5. AI 算法设计

### 5.1 难度策略（`ai/difficulty.ts`）

| 难度 | 策略 | 搜索深度 | 时间上限 |
| --- | --- | --- | --- |
| easy | 随机落子（仅避开已被占位） | 0 | < 50ms |
| normal | 启发式评分单步选择 | 1 | ≤ 500ms |
| hard | 极小极大 + Alpha-Beta 剪枝 | 3~4 层 | ≤ 1500ms |

### 5.2 棋型评分（`ai/evaluator.ts`）

对每个候选空位，分别计算「己方落子后形成的棋型得分」与「对手落子后形成的棋型得分」，取二者较大值作为该位评分（攻防一体）。

**棋型分值表**：

| 棋型 | 说明 | 分值 |
| --- | --- | --- |
| FIVE | 五连（直接获胜） | 1_000_000 |
| OPEN_FOUR | 活四（两端开放，下一手必胜） | 100_000 |
| FOUR | 冲四（一端封闭） | 10_000 |
| OPEN_THREE | 活三 | 1_000 |
| THREE | 眠三 | 100 |
| OPEN_TWO | 活二 | 100 |
| TWO | 眠二 | 10 |
| ONE | 单子 | 1 |

```typescript
// 评估单个候选位的综合得分
function evaluatePoint(board: Board, coord: Coord, player: Player): number;

// 评估整盘局势（用于搜索节点）
function evaluateBoard(board: Board, player: Player): number;
```

**评估流程**：
1. 模拟在 `coord` 落子。
2. 沿 4 方向扫描以该点为中心的连子模式，识别棋型并累加得分。
3. 还原棋盘，返回得分。

### 5.3 极小极大搜索（`ai/searcher.ts`）

```typescript
interface SearchOptions {
  depth: number;
  alpha: number;
  beta: number;
  maximize: boolean;
}

// 返回最佳落子坐标
function findBestMove(board: Board, player: Player, difficulty: Difficulty): Coord;

// 内部递归搜索
function minimax(board: Board, depth: number, alpha: number, beta: number,
                 current: Player, rootPlayer: Player): number;
```

**优化策略**：
1. **候选位剪枝**：仅考虑现有棋子周围 2 格范围内的空位，避免遍历全盘。
2. **Alpha-Beta 剪枝**：标准实现，显著减少搜索节点。
3. **走法排序**：候选位按启发式评分降序排列，提升剪枝效率。
4. **时间守卫**：搜索开始记录时间戳，超时立即返回当前最优解。
5. **必胜短路**：若发现直接五连或可阻挡的活四，立即返回，不再深入搜索。

## 6. 状态管理设计

### 6.1 Store 结构（`store/gameStore.ts`）

采用 Zustand 实现，单一 `useGameStore` 暴露状态与动作。

```typescript
interface GameStore extends GameState {
  // 动作
  startGame(config: GameConfig): void;
  place(coord: Coord): void;          // 玩家落子入口
  aiMove(): void;                     // 触发 AI 落子
  undo(): void;                       // 悔棋
  resign(): void;                     // 认输
  restart(): void;                    // 重开
  backToMenu(): void;                 // 返回主菜单
  // 派生
  getStepCount(player: Player): number;
  getElapsedMs(): number;
}
```

### 6.2 状态流转图

```
                  startGame
       idle ─────────────────► playing
        ▲                         │
        │ backToMenu              │ place/aiMove
        │                         ▼
        │                     判定胜负
        │                         │
        │            ┌────────────┼────────────┐
        │            ▼            ▼            ▼
        │          won         draw        resigned
        │            │            │            │
        └────────────┴────────────┴────────────┘
                            restart
```

**关键流转规则**：
- `idle → playing`：调用 `startGame`，初始化棋盘、历史、计时。
- `playing → playing`：合法落子且未分胜负。
- `playing → won / draw`：判定函数返回结果。
- `playing → resigned`：玩家点击认输并确认。
- `won / draw / resigned → playing`：调用 `restart`，清空棋盘但保留配置。
- `任意 → idle`：调用 `backToMenu`。

### 6.3 悔棋策略
- PVP 模式：`undoMoves(history, 1)`。
- PVE 模式：`undoMoves(history, 2)`，保证回合仍归玩家。
- 悔棋后通过 `rebuildBoard(history)` 还原棋盘，并重新计算 `currentTurn`。

### 7.1 组件树

```
App
├── MainMenu                  (status === 'idle')
│   ├── ModeSelector
```

### 7.2 关键组件职责

| 组件 | 职责 | 计算属性 |
| --- | --- | --- |
| `MainMenu` | 模式 / 执子 / 难度选择，触发 `startGame` | — |
| `GameBoard` | 渲染棋盘网格与棋子，捕获点击坐标 | 棋盘尺寸自适应计算 |
| `Stone` | 渲染单枚棋子，支持最新手高亮、连珠高亮 | — |
| `InfoPanel` | 显示回合、步数、计时，提供操作按钮 | 派生数据来自 Store |
| `ResultModal` | 对局结果展示与后续操作入口 | — |
| `ConfirmDialog` | 认输 / 返回主菜单的二次确认 | — |

### 7.3 棋盘坐标映射
- 点击事件 `e.clientX / e.clientY` → 棋盘相对坐标 → 网格交叉点索引。
- 采用 `getBoundingClientRect` 计算棋盘容器位置，单元格间距 = 棋盘宽度 / (BOARD_SIZE - 1)。
- 命中半径取半个单元格，避免边缘误触。

## 8. 关键流程时序

### 8.1 玩家落子流程（PVP）
```
玩家点击棋盘
   │
   ▼
GameBoard 派发 place(coord)
   │
   ▼
Store.place: 校验 canPlace
   │  否 → 忽略
   │  是 ↓
   ▼
board = placeStone(board, coord, turn)
history = appendMove(history, move)
   │
   ▼
result = checkWin(board, coord)
   │  胜 → status='won', winResult=result, stopTimer
   │  否 → result = checkDraw(board)
   │           和 → status='draw', stopTimer
   │           否 → currentTurn = nextTurn(turn)
   ▼
组件订阅 Store 自动重渲染
```

### 8.2 AI 落子流程（PVE）
```
玩家落子完成且未分胜负
   │
   ▼
Store 检测 mode==='pve' 且 currentTurn===aiSide
   │
   ▼
设置 'AI 思考中' 标志
   │
   ▼
异步调用 findBestMove(board, aiSide, difficulty)
   │  (使用 setTimeout/requestIdleCallback 让出主线程)
   ▼
coord = AI 结果
   │
   ▼
执行与玩家落子相同的 place 流程
   │
   ▼
清除 'AI 思考中' 标志
```

### 8.3 悔棋流程
```
玩家点击「悔棋」
   │
   ▼
Store.undo:
   n = (mode === 'pve') ? 2 : 1
   history = undoMoves(history, n)
   board = rebuildBoard(history)
   currentTurn = 末手对方的下一手方
   status 仍为 'playing'
   │
   ▼
组件重渲染
```

## 1. 设计目标与原则

### 1.1 设计目标
- 将《需求文档.md》中定义的 9 项功能需求转化为可落地的技术方案。

### 1.2 设计原则
1. **关注点分离**：棋盘逻辑、胜负判定、AI 算法、UI 渲染、状态管理各自独立模块，互不耦合。

3. **单文件可测**：关键算法不依赖 DOM 与框架，可在 Node 环境直接单测。
4. **零后端依赖**：所有状态驻留内存，偏好类信息持久化至 localStorage。
5. **类型安全**：全量 TypeScript，核心数据结构以类型别名 / 接口显式约束。

```typescript
  col: number;
```

## 4. 核心算法设计

### 4.1 棋盘初始化与落子校验（`logic/board.ts`）

```typescript
// 创建空棋盘
function createEmptyBoard(): Board;

// 判断坐标是否在棋盘范围内
function isInRange(coord: Coord): boolean;

// 判断指定位置是否可落子（在范围内且为空）
function canPlace(board: Board, coord: Coord): boolean;

// 落子（不可变更新，返回新棋盘）
function placeStone(board: Board, coord: Coord, player: Player): Board;
```

**设计要点**：
- `placeStone` 采用不可变更新（拷贝新行数组），保证 React 状态可比对、便于悔棋回放。
- 所有函数为纯函数，无副作用。

### 4.2 胜负判定（`logic/referee.ts`）

```typescript
// 判定最后一次落子是否形成五连珠
function checkWin(board: Board, lastMove: Coord): WinResult;

// 判定是否和棋（棋盘已满且无人获胜）
function checkDraw(board: Board): boolean;
```

**算法说明**：
- 仅以最新落子点为中心，沿 4 个方向（横、竖、正斜、反斜）双向延伸统计同色连子数。
- 时间复杂度 O(1)（与棋盘大小无关，仅与 WIN_COUNT 相关），远优于全盘扫描。
- 返回的 `line` 为连珠坐标序列，用于 UI 高亮。

**伪代码**：
```
checkWin(board, last):
  player = board[last.row][last.col]
  if player == 0: return { winner: null, line: [], direction: null }
  for each (dr, dc, dir) in DIRECTIONS:
    count = 1
    line = [last]
    // 正向延伸
    (r, c) = (last.row + dr, last.col + dc)
    while inRange(r,c) and board[r][c] == player:
      count++; line.push((r,c)); r += dr; c += dc
    // 反向延伸
    (r, c) = (last.row - dr, last.col - dc)
    while inRange(r,c) and board[r][c] == player:
      count++; line.unshift((r,c)); r -= dr; c -= dc
    if count >= WIN_COUNT:
      return { winner: player, line, direction: dir }
  return { winner: null, line: [], direction: null }
```

### 4.3 棋谱历史管理（`logic/history.ts`）

```typescript
// 追加一步
function appendMove(history: MoveRecord[], move: MoveRecord): MoveRecord[];

// 悔棋 N 步（N 由调用方决定：PVP=1, PVE=2）
function undoMoves(history: MoveRecord[], n: number): MoveRecord[];

// 根据历史重建棋盘（用于悔棋后还原）
function rebuildBoard(history: MoveRecord[]): Board;
```

**设计要点**：
- 悔棋后通过 `rebuildBoard` 从剩余历史重建棋盘，避免维护多份棋盘快照。
- PVE 模式悔棋两步保证回合仍归玩家。

### 4.4 回合切换（`logic/turn.ts`）

```typescript
function nextTurn(current: Player): Player;   // 1 -> 2, 2 -> 1
function isFirstMove(history: MoveRecord[]): boolean;
```

## 5. AI 算法设计

### 5.1 难度策略（`ai/difficulty.ts`）

| 难度 | 策略 | 搜索深度 | 时间上限 |
| --- | --- | --- | --- |
| easy | 随机落子（仅避开已被占位） | 0 | < 50ms |
| normal | 启发式评分单步选择 | 1 | ≤ 500ms |
| hard | 极小极大 + Alpha-Beta 剪枝 | 3~4 层 | ≤ 1500ms |

### 5.2 棋型评分（`ai/evaluator.ts`）

对每个候选空位，分别计算「己方落子后形成的棋型得分」与「对手落子后形成的棋型得分」，取二者较大值作为该位评分（攻防一体）。

**棋型分值表**：

| 棋型 | 说明 | 分值 |
| --- | --- | --- |
| FIVE | 五连（直接获胜） | 1_000_000 |
| OPEN_FOUR | 活四（两端开放，下一手必胜） | 100_000 |
| FOUR | 冲四（一端封闭） | 10_000 |
| OPEN_THREE | 活三 | 1_000 |
| THREE | 眠三 | 100 |
| OPEN_TWO | 活二 | 100 |
| TWO | 眠二 | 10 |
| ONE | 单子 | 1 |

```typescript
// 评估单个候选位的综合得分
function evaluatePoint(board: Board, coord: Coord, player: Player): number;

// 评估整盘局势（用于搜索节点）
function evaluateBoard(board: Board, player: Player): number;
```

**评估流程**：
1. 模拟在 `coord` 落子。
2. 沿 4 方向扫描以该点为中心的连子模式，识别棋型并累加得分。
3. 还原棋盘，返回得分。

### 5.3 极小极大搜索（`ai/searcher.ts`）

```typescript
interface SearchOptions {
  depth: number;
  alpha: number;
  beta: number;
  maximize: boolean;
}

// 返回最佳落子坐标
function findBestMove(board: Board, player: Player, difficulty: Difficulty): Coord;

// 内部递归搜索
function minimax(board: Board, depth: number, alpha: number, beta: number,
                 current: Player, rootPlayer: Player): number;
```

**优化策略**：
1. **候选位剪枝**：仅考虑现有棋子周围 2 格范围内的空位，避免遍历全盘。
2. **Alpha-Beta 剪枝**：标准实现，显著减少搜索节点。
3. **走法排序**：候选位按启发式评分降序排列，提升剪枝效率。
4. **时间守卫**：搜索开始记录时间戳，超时立即返回当前最优解。
5. **必胜短路**：若发现直接五连或可阻挡的活四，立即返回，不再深入搜索。

## 6. 状态管理设计

### 6.1 Store 结构（`store/gameStore.ts`）

采用 Zustand 实现，单一 `useGameStore` 暴露状态与动作。

```typescript
interface GameStore extends GameState {
  // 动作
  startGame(config: GameConfig): void;
  place(coord: Coord): void;          // 玩家落子入口
  aiMove(): void;                     // 触发 AI 落子
  undo(): void;                       // 悔棋
  resign(): void;                     // 认输
  restart(): void;                    // 重开
  backToMenu(): void;                 // 返回主菜单
  // 派生
  getStepCount(player: Player): number;
  getElapsedMs(): number;
}
```

### 6.2 状态流转图

```
                  startGame
       idle ─────────────────► playing
        ▲                         │
        │ backToMenu              │ place/aiMove
        │                         ▼
        │                     判定胜负
        │                         │
        │            ┌────────────┼────────────┐
        │            ▼            ▼            ▼
        │          won         draw        resigned
        │            │            │            │
        └────────────┴────────────┴────────────┘
                            restart
```

**关键流转规则**：
- `idle → playing`：调用 `startGame`，初始化棋盘、历史、计时。
- `playing → playing`：合法落子且未分胜负。
- `playing → won / draw`：判定函数返回结果。
- `playing → resigned`：玩家点击认输并确认。
- `won / draw / resigned → playing`：调用 `restart`，清空棋盘但保留配置。
- `任意 → idle`：调用 `backToMenu`。

### 6.3 悔棋策略
- PVP 模式：`undoMoves(history, 1)`。
- PVE 模式：`undoMoves(history, 2)`，保证回合仍归玩家。
- 悔棋后通过 `rebuildBoard(history)` 还原棋盘，并重新计算 `currentTurn`。

## 8. 关键流程时序

### 8.1 玩家落子流程（PVP）
```
玩家点击棋盘
   │
   ▼
GameBoard 派发 place(coord)
   │
   ▼
Store.place: 校验 canPlace
   │  否 → 忽略
   │  是 ↓
   ▼
board = placeStone(board, coord, turn)
history = appendMove(history, move)
   │
   ▼
result = checkWin(board, coord)
   │  胜 → status='won', winResult=result, stopTimer
   │  否 → result = checkDraw(board)
   │           和 → status='draw', stopTimer
   │           否 → currentTurn = nextTurn(turn)
   ▼
组件订阅 Store 自动重渲染
```

### 8.2 AI 落子流程（PVE）
```
玩家落子完成且未分胜负
   │
   ▼
Store 检测 mode==='pve' 且 currentTurn===aiSide
   │
   ▼
设置 'AI 思考中' 标志
   │
   ▼
异步调用 findBestMove(board, aiSide, difficulty)
   │  (使用 setTimeout/requestIdleCallback 让出主线程)
   ▼
coord = AI 结果
   │
   ▼
执行与玩家落子相同的 place 流程
   │
   ▼
清除 'AI 思考中' 标志
```

### 8.3 悔棋流程
```
玩家点击「悔棋」
   │
   ▼
Store.undo:
   n = (mode === 'pve') ? 2 : 1
   history = undoMoves(history, n)
   board = rebuildBoard(history)
   currentTurn = 末手对方的下一手方
   status 仍为 'playing'
   │
   ▼
组件重渲染
```

## 14. 风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| AI 困难难度搜索耗时超限 | 卡顿、体验差 | 时间守卫 + 动态降深 + 候选位剪枝 |
| 高分辨率下棋盘点击坐标偏差 | 落子位置错误 | 使用 getBoundingClientRect 精确换算，命中半径取半格 |
| 移动端误触 | 错落子 | 落子前显示预览子，点击确认（移动端可启用） |
| 浏览器禁用 localStorage | 主题不保持 | storage 封装 try-catch，降级为内存默认值 |
